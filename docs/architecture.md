# 嵌套解压助手设计

> 文档状态：本文同时记录完整产品设计和实施进度。当前已完成 M0 至 M3，M4 已完成主要流程；任务队列、多选启动、设置、历史元数据、本机明文密码池和逐步任务日志已实现，M5 的断点恢复和汇总报告尚未实现。

## 1. 文档目的

本文定义“嵌套解压助手”的第一版架构、安全边界和验收标准。

该工具面向 Windows 本地文件处理。一个根文件可能被错误命名、藏在可播放 MP4/TXT 等前置外壳中、被 Apate 伪装，并在多层解压后仍包含压缩包。程序可为任务提供单独的优先密码，并按顺序尝试用户保存的常用密码池；所有候选均失败时暂停等待用户补充。

本文不包含可执行代码，也不授权程序修改、删除或移动任何源文件。

## 2. 产品目标

用户选择一个或多个根文件，按需设置任务密码或长期密码池，并设置统一的“输出目录”后，程序应当：

1. 通过本地桌面界面创建、查看和恢复任务；
2. 在独立工作目录中处理根文件副本；
3. 先按当前文件名交给 WinRAR 测试；
4. 直接测试失败、但文件头明确属于 ZIP、RAR 或 7Z 时，在工作目录生成一份正确扩展名的副本重试；
5. 正确后缀副本仍失败时，检查是否存在结构完整的嵌入式 ZIP，并在工作目录生成去外壳副本；
6. 前述路线都失败时，对工作副本执行 Apate 兼容还原；
7. 对还原结果再次执行 WinRAR 测试；
8. 成功解压后，对解压结果中的候选归档递归执行同一流程；
9. 按“任务密码 → 默认密码 → 其他密码池成员 → 无密码”尝试，全部失败时要求用户补充；
10. 将最终文件放入统一输出根目录下的任务子目录，并记录逐步日志和最终结果。

## 3. 非目标

第一版明确不做以下事情：

- 不批量扫描整个下载目录；
- 不修改、重命名、删除或移动源文件；
- 不自动执行解压出的 EXE、脚本、快捷方式或安装程序；
- 不删除源文件；成功后清理可重新生成的工作副本，失败后只保留小型失败说明；
- 不自动猜测、爆破或枚举密码；
- 不使用 7-Zip 作为解压引擎；
- 不通过窗口点击或拖拽方式操控 apate.exe；
- 不采用浏览器页面、远程服务或 Web 管理后台；
- 不提供网络上传、云同步、远程任务或共享功能；
- 不做媒体归类、内容识别或 NAS 整理。

## 4. 核心判断

### 4.1 改扩展名与 Apate 还原不是同一件事

单纯把文件名改成 ZIP、RAR 或 7Z 后缀，不会改变文件本身的格式字节。

Apate 伪装会覆盖文件开头的真实格式字节，并把原始文件头及面具长度写入文件尾部。还原后才会恢复真实格式字节并截掉追加内容。因此，出现“手动改后缀解压失败，Apate 还原后成功”的情况是正常的。

### 4.2 Apate 还原不能直接作用于原文件

Apate 的公开实现会原地覆盖、截断并重命名处理文件，且没有可可靠验证“文件一定经过 Apate 伪装”的标记。对普通文件误执行还原可能造成不可恢复损坏。

本项目的 Apate 兼容模块必须始终遵守：

- 原文件只读；
- M0 直接从只读输入流生成独立输出副本，不在输入文件上执行写入、截断或改名；
- 完整任务流程由 JobCoordinator 先建立工作副本，再把工作副本交给 ApateRestoreAdapter；
- 只有还原结果通过 WinRAR 归档测试后才视为成功；
- 失败副本可留作人工排查，但绝不影响源文件。

### 4.3 改后缀、嵌入式 ZIP 与 Apate 是三条不同路线

- **直接改后缀型**：文件从第一个字节开始就是 ZIP/RAR/7Z，只是名字写成 MP4、TXT 等；程序先让 WinRAR 直接识别，失败后才根据正式文件头生成一份正确后缀的工作副本，不修改源文件名；
- **嵌入式 ZIP 型**：文件开头是真实可播放外壳，ZIP 从中间某个偏移开始；程序必须校验 ZIP 的结束目录、中央目录和本地文件头，再剥离外壳；
- **Apate 型**：真实归档头被面具覆盖，原始头部倒序保存在文件尾部，必须按 Apate 布局还原。

三者不能互相替代。普通 MP4 中偶然出现 `PK` 字节不代表藏有 ZIP，只有完整目录关系成立才进入嵌入式路线。

当前兼容边界：

- 外层文件名可以是 MP4、TXT、PDF、JPG 或其他扩展名；识别器不按扩展名决定是否存在嵌入式 ZIP；
- 完整伪装文件若随后按原始字节切成 `name.zip.001/.002/.003`，程序会先按编号拼接，再识别外壳；
- 若首卷单独改名为 `name.mp4` 或 `name.txt`、其余卷仍叫 `name.002/.003`，并且同目录只有一个同名首卷候选、编号从 `.002` 连续排列，程序会自动归组并在工作目录拼接；候选不唯一或缺卷时拒绝猜测；
- 若先分卷、再只伪装首卷或分别伪装每一卷，必须先逐卷还原再按正确顺序组合，当前任务模型尚不支持；
- 嵌入式结构检测当前只解析普通 ZIP 中央目录，不解析 ZIP64，也不剥离带任意前置外壳的 RAR/7Z；这些格式若从第一个字节开始就是归档，或采用受支持的 Apate 布局，仍可处理；
- 当前只扫描文件末尾 8 MB 寻找 ZIP 结束目录，尾随数据超过该范围的特殊文件需要后续增加深度扫描模式。

交互原则：上述路线必须由程序自动依次验证。用户只负责添加文件、按需提供密码和查看结果，不需要先判断文件属于“改后缀、正确后缀副本、嵌入式 ZIP、Apate”中的哪一种。对于不规则分卷，程序应在工作目录尝试可证明的候选顺序，并以 WinRAR 是否产生有效文件作为判据；仅当不存在唯一有效组合时才进入人工确认。

### 4.4 统一输出目录必须按任务隔离

所有最终文件应输出至：

~~~text
  <输出根目录>/
  <任务标识>/
~~~

不能把不同任务的最终文件平铺在同一个目录中，否则会出现同名文件覆盖。

## 5. 用户流程

~~~text
创建任务
  -> 选择或拖入一个或多个根文件，每个文件成为独立任务
  -> 通常使用设置中的长期密码池
  -> 特殊任务可双击该行密码栏填写优先密码，留空则使用密码池
  -> 在设置中确认输出目录、工作目录、WinRAR 和可选资源限制
  -> 启动当前任务、所选等待任务或全部等待任务

处理当前文件
  -> 创建或定位工作副本
  -> 按当前文件名执行 WinRAR 测试
  -> 直接失败且正式文件头匹配：生成一份正确后缀工作副本并重试
  -> 任一路线成功：解压到当前层目录
  -> 仍失败：检查并剥离结构完整的嵌入式 ZIP
  -> 仍失败：执行 Apate 兼容还原副本
  -> 对还原结果再次执行格式尝试和 WinRAR 测试
  -> 仍失败：
       根文件：标记为“无法识别”，任务暂停
       内层文件：保留为最终文件或“待人工判断”文件

解压成功
  -> 扫描新产生的文件
  -> 发现候选归档：进入下一层，按同一候选顺序寻找可用密码
  -> 所有候选均失败：任务暂停，等待用户输入
  -> 非归档最终文件：复制到任务结果目录

结束
  -> 更新表格状态和本地历史元数据
  -> 写入逐步文本日志和 JSONL 事件日志
  -> 成功后清理工作副本；失败后仅保留 failure.json
~~~

## 6. 架构

当前采用“多任务队列、任务目录隔离”的架构。默认一次运行一个任务；用户可在设置中改为同时运行 1–5 个任务。每个任务拥有独立密码、工作目录和输出目录，不能共享可变任务状态。

~~~text
Tkinter 本地桌面界面
        |
        v
任务队列 QueueScheduler
        |
        +-- 历史元数据 HistoryStore（不含密码）
        +-- 设置 SettingsStore
        |
        v
独立任务 JobCoordinator
        |
        +-- 当前任务内存密码
        +-- 本机明文密码池 PasswordVault
        +-- 工作目录 WorkArea
        +-- 格式尝试 ArchiveProbe
        |       |
        |       +-- WinRARAdapter
        |       +-- EmbeddedZipExtractor
        |
        +-- ApateRestoreAdapter
        |
        +-- 递归调度 LayerScheduler
        |
        +-- 最终输出 Finalizer
        |
        +-- 本地日志 JobReporter
~~~

### 6.0 本地桌面界面

第一版采用 Tkinter 桌面界面，不采用浏览器页面。

技术选择：

| 方案 | 优点 | 局限 | 结论 |
|---|---|---|---|
| Tkinter | Python 自带，无额外依赖，当前 Windows 环境可直接运行 | 复杂任务表格和层级日志的维护成本较高 | 采用 |
| PySide6 | 适合复杂任务表格、进度和模态密码输入 | 当前环境无法取得第三方依赖，且第一版不需要其复杂控件 | 后续可按需迁移 |

当前界面包含五个工作区域：

1. 顶部菜单和工具栏：添加多个文件、开始所选或全部等待任务、重试、移除和打开结果；
2. 任务表格：显示文件、状态、当前处理阶段、密码策略、结果目录和创建时间；
3. 行级密码编辑器：双击密码栏后直接在该单元格内保存或更换内存密码；
4. 状态栏：显示队列数量和拖拽是否可用；
5. 设置窗口：配置 WinRAR.exe、默认工作目录、输出目录、可选资源限制和 1–5 个同时运行任务。

界面支持任务多选、右键操作、文件多选、文件夹第一层展开和 Windows 原生文件/文件夹拖拽，不需要第三方 Python 包。文件夹导入不递归，避免意外把整棵目录树中的文件全部创建为任务。按用户明确选择，任务密码编辑器和密码池列表均直接显示完整密码，方便核对录入内容；使用密码池的任务行只显示候选数量。表格进度是处理阶段文字，不是无法可靠预估的百分比。

程序通过 `ui/app_assets.py` 统一定位界面资源：源码运行时从项目 `assets/` 读取，PyInstaller 打包后从 `_MEIPASS/assets/` 读取。`app.ico` 提供 16 至 256 像素的多尺寸 Windows 图标，用于主窗口、任务栏、子窗口和已打包 EXE；`app-logo.png` 用于“关于”窗口。本次不引入界面背景图片。

Windows 发布采用 PyInstaller `onedir` 便携目录并整体压缩为 ZIP，而不是只发布一个孤立 EXE。当前 Conda Python 的 Tcl/Tk 在 PyInstaller 设置 `TCL_LIBRARY`、`TK_LIBRARY` 后无法正常初始化，因此 `packaged_runtime.py` 在导入 Tkinter 前清除这两个打包环境变量，构建脚本同时保留程序目录旁的 `lib/tcl8.6` 与 `lib/tk8.6`。用户必须完整解压 ZIP，不能只移动其中的 EXE。源码运行时状态保存在项目 `runtime/`；打包程序的设置、历史和日志保存在 `%LOCALAPPDATA%\NestedExtractionAssistant\`，不会写回发布目录。

### 6.1 JobCoordinator

职责：

- 创建任务；
- 保存任务状态；
- 调度每一层的处理顺序；
- 在密码错误、资源限制或不可识别文件时暂停；
- 把结果交回队列状态模型；
- 保证每个任务的工作区和密码互相隔离。

不负责：

- 直接读写归档格式；
- 直接处理 Apate 文件字节；
- 维护或同步密码池持久化格式。

### 6.2 ArchiveProbe 与 WinRARAdapter

ArchiveProbe 负责判断当前文件是否为归档。WinRARAdapter 负责以无窗口、无 shell 的方式调用已安装的 WinRAR 进行只读测试和解压。

WinRAR 7.0.0 的本机合成测试已经证明：有效 ZIP 即使改名为 `.mp4`，`t` 和 `x` 命令仍能按文件内容直接识别。因此当前实现不再为 ZIP、RAR、7Z 分别复制整份大文件并改后缀。这样可避免对大文件产生三份无意义工作副本。

当前固定识别顺序：

~~~text
1. WinRAR 直接测试当前文件；
2. 归档测试只用于判断文件是否可能可解压；实际解压时依次使用每个用户候选，并为每次尝试建立独立临时目录；
3. 候选返回密码错误或成功但输出为空时清理本次临时目录并继续，只有确实产生文件才停止尝试；
4. 常规测试失败后，校验文件尾部的 ZIP 结束目录，并反推中央目录、前置外壳长度和本地文件头位置；
5. 结构关系完整时只复制 ZIP 字节区间，并让 WinRAR 测试该工作副本；
6. 嵌入式 ZIP 仍不匹配时，只读取少量头尾字节检查 Apate 还原结果是否可能具有 ZIP、RAR 或 7Z 文件头；
7. 只有通过预判时才生成独立 Apate 还原副本；
8. WinRAR 再次测试还原副本；
9. 仍未识别的内层文件作为最终文件处理。
~~~

当前界面显示任务状态、处理阶段和最终输出。队列历史元数据已持久化；每次运行还会追加文本日志和 JSONL 事件日志，记录开始、每个可见处理阶段、异常堆栈和最终统计。日志目前记录：

- 源路径、工作路径和输出路径；
- 直接识别、正确后缀副本、嵌入式 ZIP、Apate 和分卷拼接等处理阶段；
- 最终状态、处理方式、文件数、容量、压缩包数和分卷组数；
- 未预期异常类型和堆栈；按用户明确选择，任务密码、密码池和实际候选也写入本机日志。

日志不等于可恢复检查点。按层恢复和面向用户的汇总报告仍属于 M5 后续工作。

#### 候选文件范围

内层文件可能使用 TXT、MP4 或任意其他名称，不能只按扩展名判断。这里存在两种可选策略：

| 策略 | 做法 | 优点 | 风险 |
|---|---|---|---|
| 保守模式 | 只处理已知归档后缀和明显归档文件头 | 速度快、工作副本少 | 会漏掉改名或伪装后的内层压缩包 |
| 穷尽模式 | 对当前层的每个常规文件都在工作副本上尝试归档格式，再进入 Apate 分支 | 能覆盖未知后缀和伪装文件 | 时间、临时空间和 WinRAR 调用次数更多 |

第一版推荐“单任务穷尽模式”，因为产品目标就是处理文件名不可信的嵌套压缩包。它只针对用户主动选择的单个根任务，不用于批量扫描目录。

当前采用经过真实样本验证的分层模式：每个内层文件先交给 WinRAR 直接测试；失败后只读取开头 8 字节，若正式签名明确且后缀错误，只生成一份正确后缀副本；再检查嵌入式 ZIP 结构，只生成一份去外壳副本；最后才进入 Apate 分支。2026-07-17 的三个真实 MP4 样本证明 WinRAR 直接读取 `.mp4` 路径不会自动跳过其视频外壳，但 ZIP 目录可以稳定反推出 14 MB、22 MB 和 62 MB 的外壳边界。对第一条样本的进一步复核显示，完整文件复制为 `.zip` 后 WinRAR 也能解出内层文件；当前仍优先剥离外壳，因为带真实 MP4 前缀的文件不会通过“开头正式签名”条件。程序不会盲目复制 ZIP/RAR/7Z 三份大文件，也不要求安装 Bandizip。

#### 多分卷归档

多分卷归档不是多个独立压缩包。ArchiveProbe 必须先调用 VolumeResolver，把同一目录中的关联文件识别为一个分卷组，再从正确首卷启动 WinRAR。

分卷规则：

1. 对“同一基础名称 + 连续数字”的自定义分卷，例如 resource.0.zip、resource.1.zip、resource.2.zip，优先从数字最小的卷开始；
2. 对 WinRAR 的标准 part 命名，优先从 part 编号最小的卷开始；
3. 对 ZIP 的标准分卷命名，不能机械按最小数字启动；应交由规则识别器选择 ZIP 主文件；
4. 对 `name.7z.001 / .002 / .003` 这类原始字节切分卷，在独立工作目录按编号流式拼接为 `name.7z`，再交给 WinRAR；
5. 对 `name.mp4 + name.002 + name.003` 一类不规则命名，仅在同目录存在唯一同名首卷候选且编号连续时自动拼接；
6. 无法确定首卷时，任务应进入 awaiting_volume_selection 状态；该人工选择界面尚未实现，当前会阻止任务并说明原因；
7. 所有分卷必须保持在同一目录并保留原始命名关系，拼接只生成工作副本；
8. 如果一个分卷组进入 Apate 分支，程序必须记录哪些卷生成了还原副本，不得把单个分卷误当成完整归档。

这项规则避免把分卷逐个解压，也避免依赖 WinRAR 对原始字节切分卷不可靠的返回码。拼接前必须验证编号连续、总大小不超过任务限制且工作磁盘空间充足；拼接失败时删除不完整的 `.joining` 工作文件，源卷不受影响。

WinRARAdapter 是外部依赖风险点。实现前必须在用户实际安装的 WinRAR 上用无私密测试文件验证：

1. 测试命令是否能返回可识别的成功、格式错误和密码错误；
2. 错误扩展名工作副本是否能按预期被测试；
3. 密码传递方式是否满足自动处理要求；
4. 解压输出是否可限制在指定任务目录内。
5. 使用 WinRAR.exe 的后台模式时，任务进度、退出状态和密码错误能否被本程序可靠识别。

### 6.3 ApateRestoreAdapter

职责：在独立输出副本中复现 Apate 的“还原”文件格式。

输入：

- 待还原工作副本；
- 独立还原输出路径；
- 文件大小与资源限制。

输出状态：

~~~text
restored_candidate
not_plausible
restore_failed
resource_limit_reached
~~~

重要限制：

- restored_candidate 只代表已产生还原副本，不代表它一定是归档文件；
- Apate 文件格式没有适合本工具使用的强校验标记，不能仅凭文件尾部数据确信它一定经过伪装；
- 只有后续 WinRAR 归档测试成功，任务才记录为“Apate 还原成功”；
- Apate 兼容模块不直接修改输入文件。

输出命名规则：

~~~text
输入：sample.rar.mp4
还原副本：sample.rar

输入：sample.mp4
还原副本：sample
~~~

还原副本仍将进入 ZIP、RAR、7Z 候选格式尝试，不能假设其真实格式。

### 6.4 LayerScheduler

职责：

- 维护压缩包嵌套层级；
- 继承任务密码；
- 将每个成功解压结果中的候选文件加入待处理队列；
- 不设置固定最大层数，以“没有可继续处理的归档”为正常结束条件；
- 使用用户在界面中设置的最大文件数和最大累计输出容量控制资源风险；
- 防止同一文件被重复处理。

默认策略：

- 按“当前层级优先”处理；
- 每个文件最多执行一次常规格式尝试和一次 Apate 分支；
- 同一文件的同一候选后缀不重复尝试；
- 触及用户设置的资源限制时立即暂停任务，由用户选择继续或停止；
- 不因达到固定层数而停止任务。

### 6.5 PasswordVault

PasswordVault 按用户明确选择，将密码池以可直接核对的明文 JSON 保存在 `runtime/state/password-vault.json`。保存时先写临时文件再替换正式文件，避免写入中断留下半份 JSON。旧版 DPAPI 或 Windows 凭据索引不再读取，界面会要求用户重新保存一次。

密码规则：

- 密码池第一项为默认密码，最多保存 50 个，可按换行或 `|` 批量添加；
- 单任务优先密码只在内存中存在，不写入历史；
- 每个压缩包按“任务密码、密码池顺序、无密码”尝试；
- 实际解压返回密码错误时继续下一项；若 WinRAR 返回成功但没有产生文件，也视为该候选未匹配并继续；格式无效或系统错误不会盲目重试；
- 用户池全部失败后进入 `password_required`，不自动猜测、生成或爆破密码；
- 运行进度和任务日志明文记录每个实际密码候选，方便核对失败顺序；历史摘要和 `failure.json` 不重复保存密码。

WinRARAdapter 的实际验证确认，无人值守解压需要通过命令行参数传递当前候选，因此运行期间密码会短暂出现在本机进程参数中。密码池本身也按用户选择明文保存，程序不再声称密码在本机上被加密隐藏。

WinRAR.exe 可以通过命令行后台处理归档，但后台模式仅表示最小化到系统托盘，不等同于没有密码或错误提示。正常流程应由本程序界面处理密码输入和任务状态，不能依赖 WinRAR 的独立弹窗。

### 6.6 WorkArea 与 Finalizer

建议目录：

~~~text
<工作根目录>/
  jobs/
    <任务标识>/
      task.json
      input/
      candidates/
      apate-restored/
      layers/
        000/
        001/
      reports/

<输出根目录>/
  <任务标识>/

<程序目录>/runtime/logs/
  all-tasks.log
  all-task-events.jsonl
  <队列任务标识>/
    task.log
    task-events.jsonl
~~~

规则：

- input、candidates 和 apate-restored 只保存工作副本；
- layers 保存每层的 WinRAR 输出；
- 任务结果目录保存不可继续解压的最终文件；
- 最终输出保留相对层级路径，避免同名覆盖；
- 工作目录和输出目录都不能位于源文件所在目录内；
- 成功结果完成移动和容量核对后，自动清理可重新生成的工作副本；
- 失败任务清理大体积临时文件，只在原任务工作目录保留不含密码的 `failure.json`；
- 不自动清理旧版本已经产生的目录，避免升级时擅自删除用户现有数据。
- 任务日志独立于成功后会清理的工作目录，以便后续排错；日志只追加，并按用户选择明文保存密码候选。

## 7. 任务状态模型

当前队列表格使用以下外层状态：

~~~text
pending
running
completed
password_required
not_archive
blocked
failed
interrupted
~~~

`interrupted` 表示程序上次关闭时任务尚未结束，只用于提示历史状态，不表示可以断点续作。下面是完整产品设计中的任务内部阶段；当前界面把这些阶段作为进度文字展示，没有全部做成可恢复状态。

~~~text
created
preparing
testing_archive
restoring_apate_copy
extracting
scanning_layer
awaiting_password
awaiting_volume_selection
awaiting_decision
completed
failed
stopped
~~~

关键状态转换：

~~~text
testing_archive
  -> extracting                 归档测试成功
  -> restoring_apate_copy       常规格式全部失败

restoring_apate_copy
  -> testing_archive            已生成还原副本
  -> awaiting_decision          还原不可信或资源受限

extracting
  -> scanning_layer             解压成功
  -> awaiting_password          密码错误
  -> failed                     解压器报告不可恢复错误

scanning_layer
  -> testing_archive            发现下层候选归档
  -> awaiting_volume_selection  分卷首卷无法确定
  -> completed                  无待处理候选
~~~

## 8. 安全与可靠性要求

### 8.1 源文件保护

- 源文件只读；
- 源目录内不创建报告、临时文件或输出目录；
- 所有格式改名只发生在工作副本；
- 所有嵌入式 ZIP 剥离只写入独立工作副本；
- 所有 Apate 还原只写入独立工作副本；
- 禁止任何自动删除命令。

### 8.2 解压安全

- 解压目标目录必须是任务层目录；
- 解压完成后验证输出路径仍位于任务目录中，拒绝路径穿越；
- 不执行任何解压出的文件；
- 不设置固定嵌套深度；文件数、累计输出容量和压缩包数量默认不限制，用户可分别启用上限；
- 解压器失败时保留小型失败说明，不保留可重新生成的大体积中间文件。
- 仅使用 WinRAR.exe 的后台处理能力；不通过模拟鼠标或键盘操控其图形界面。

### 8.3 密码可见范围

- 密码池正文写入本机 `password-vault.json`，并在密码池界面完整显示；
- 任务密码在任务密码栏完整显示，但不写入任务历史；
- 不在命令历史中保存密码；
- 任务优先密码只在任务对象内存中存在，任务结束后清除；
- 运行进度和任务日志明文记录密码；任务历史摘要、失败报告和结果目录不额外保存密码；
- 公开仓库不得包含实际密码、本地凭据或私人任务历史。

### 8.4 报告隐私

任务日志和报告可能包含私有文件名和目录结构，必须默认排除在 Git 之外。公开项目只能提交虚构样例，不提交真实任务日志、报告、密码档案、工作目录或输出目录内容。
